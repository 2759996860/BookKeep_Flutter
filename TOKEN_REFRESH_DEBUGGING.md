# Token 刷新问题诊断指南

## 问题描述
**现象**：RefreshToken 未过期，但应用仍然跳转到登录页面。

## 已实施的修复

### 1. 优化并发锁等待机制
- ✅ 将等待时间从 10 秒增加到 30 秒（60次 * 500毫秒）
- ✅ 等待结束后验证 Token 是否真的刷新成功
- ✅ 添加详细的验证日志

### 2. 增强日志输出
- ✅ `isLoggedIn()` 方法中添加刷新前后的详细对比
- ✅ `refreshToken()` 方法中增加异常类型识别
- ✅ 所有关键步骤都添加了明确的标记和提示

### 3. 改进异常处理
- ✅ 区分超时、网络错误、SSL错误等不同异常类型
- ✅ 每种异常都提供可能的原因和解决建议
- ✅ 确保异常时正确释放刷新锁

## 诊断步骤

### 第一步：重现问题并收集日志

当问题发生时，请复制**完整的控制台日志**，特别关注以下内容：

```
========== 应用启动 - Token检查开始 ==========
📋 原始Token数据:
   AccessToken: ...
   RefreshToken: ...
   ExpiresTime: ...
⏰ 时间信息:
   当前时间: ...
   过期时间: ...
   是否过期: ...
   
🔍 开始检查登录状态...
📋 Token状态检查:
   AccessToken: ...
   RefreshToken: ...
   过期时间: ...
```

### 第二步：分析日志中的关键信息

#### 场景 A：AccessToken 未过期但仍跳转登录
**日志特征**：
```
✅ Token有效，剩余时间: XX小时XX分钟XX秒
```
但仍然跳转登录页。

**可能原因**：
1. 其他地方的逻辑错误（非 `isLoggedIn()` 调用）
2. 某个接口返回 401 且刷新失败
3. 手动调用了 `clearTokens()`

**排查方法**：
搜索日志中是否有以下关键词：
- `❌ Token刷新失败`
- `清除Token`
- `登录已过期，请重新登录`

---

#### 场景 B：AccessToken 过期，刷新失败
**日志特征**：
```
⚠️ AccessToken已过期
✅ RefreshToken存在，准备刷新...
🔄 ========== Token刷新流程开始 ==========
❌ Token刷新异常: ...
```

**可能原因及解决方案**：

##### B1. 网络超时
```
❌ Token刷新异常: TimeoutException
💡 提示: 刷新请求超时（30秒）
可能原因:
- 网络连接不稳定
- 服务器响应慢
- 防火墙阻止
```
**解决**：检查网络连接，增加超时时间或优化服务器性能。

##### B2. 无法连接服务器
```
❌ Token刷新异常: SocketException
💡 提示: 无法连接到服务器
可能原因:
- API地址配置错误
- 网络未连接
- 服务器未启动
```
**解决**：检查 `AppConfig.baseUrl` 配置，确认服务器运行状态。

##### B3. RefreshToken 无效（401）
```
❌ HTTP请求失败: 状态码 401
可能原因:
- RefreshToken已过期或无效
```
**解决**：虽然你认为 RefreshToken 未过期，但服务器可能认为它已失效。检查服务器的 Token 过期策略。

##### B4. 并发刷新等待超时
```
⚠️ Token正在刷新中，等待其他请求完成...
❌ 等待超时（30秒），可能存在死锁
```
**解决**：这种情况说明有多个请求同时触发刷新。检查是否有地方在循环调用需要认证的接口。

##### B5. JSON 解析失败
```
❌ 解析响应JSON失败: FormatException
```
**解决**：服务器返回的数据格式不正确。检查后端接口返回。

---

#### 场景 C：刷新成功但仍跳转登录
**日志特征**：
```
✅ Token刷新成功，视为已登录
🔍 验证新Token:
   新AccessToken: ✅ 已保存 (XXX字符)
   新RefreshToken: ✅ 已保存 (XXX字符)
   新过期时间: ✅ 2026-04-17 21:00:00.000
```
但仍然跳转登录页。

**可能原因**：
1. **严重 Bug**：`isLoggedIn()` 返回 `true`，但其他地方强制跳转
2. **导航上下文问题**：`_contextForNavigation` 无效
3. **多个地方检查 Token**：某处检查失败导致跳转

**排查方法**：
1. 搜索日志中是否有 `最终检查结果: ✅ 已登录`
2. 检查是否有 `无法导航到登录页：上下文无效`
3. 查看是否有多个 `Token检查开始` 日志

## 常见陷阱与解决方案

### 陷阱 1：应用启动时的竞态条件
**问题**：应用启动时，多个组件同时初始化，触发多次 Token 检查。

**解决**：
- 确保只在 `main.dart` 的 `_checkTokenAndNavigate()` 中检查一次
- 其他页面不要重复检查

### 陷阱 2：后台任务触发刷新
**问题**：后台定时任务或推送通知触发了需要认证的请求，导致意外刷新。

**解决**：
- 检查是否有定时器、Stream 订阅等后台任务
- 为后台任务添加独立的错误处理

### 陷阱 3：手动清除 Token
**问题**：某处代码意外调用了 `ApiService.clearTokens()`。

**解决**：
搜索代码中所有 `clearTokens()` 调用：
```bash
grep -r "clearTokens" lib/
```

### 陷阱 4：导航上下文丢失
**问题**：`_contextForNavigation` 为 null 或已销毁。

**解决**：
检查日志中是否有：
```
无法导航到登录页：上下文无效
```

## 调试技巧

### 1. 添加断点
在以下位置添加断点：
- `ApiService.isLoggedIn()` 的返回值
- `ApiService.refreshToken()` 的返回值
- `ApiService._navigateToLogin()` 的调用处

### 2. 监控 Token 变化
在 `saveTokens()` 和 `clearTokens()` 中添加日志：
```dart
print('💾 saveTokens 被调用，堆栈跟踪:');
print(StackTrace.current);
```

### 3. 模拟 Token 过期
手动修改 SharedPreferences 中的过期时间，测试刷新逻辑：
```dart
// 在调试控制台中执行
final prefs = await SharedPreferences.getInstance();
await prefs.setInt('expires_time', DateTime.now().millisecondsSinceEpoch - 1000);
```
